시니어 크로스플랫폼 개발자로서 제안하신 1차 계획안을 기술적 타당성과 1인 개발 효율성 관점에서 검토한 결과입니다.

### 쟁점 1. RN+Expo가 정말 최선인가?
[결론] **현 상황에서 최적의 선택.**
[근거] TS를 학습 중인 Python 전문가에게 Dart(Flutter)나 Swift 학습은 리소스 낭비임. Expo Config Plugins의 발전으로 위젯 등 네이티브 대응도 충분히 가능하며 개발 생산성이 가장 높음.

### 쟁점 2. ML Kit Korean 정확도 vs Apple Vision + ML Kit 분리 운영
[결론] **iOS 우선이라면 Apple Vision Framework 활용 권장.**
[근거] iOS 네이티브 Vision API는 한글 인식률이 매우 높고 무료임. Expo에서 `expo-vision-camera` 등을 통해 iOS는 Apple Vision, Android는 ML Kit으로 분기 처리하는 것이 사용자 경험 품질 면에서 월등함.

### 쟁점 3. GPT-4o vision 폴백이 정말 필요한가?
[결론] **MVP에서는 제외, 추후 유료화 모델로 검토.**
[근거] 정갈한 폰트의 문구는 온디바이스 OCR로도 99% 해결됨. 비용 발생 및 API 레이턴시를 고려할 때, 손글씨나 복잡한 배경 등 특수 케이스를 위한 'Pro' 기능으로 빼는 것이 경제적임.

### 쟁점 4. TTS 하이브리드(expo-speech + OpenAI) vs 단일 선택
[결론] **expo-speech 단일 사용으로 시작.**
[근거] 짧은 문장 위주이므로 시스템 기본 음성으로도 충분함. OpenAI TTS는 오디오 파일 캐싱, 네트워크 처리 등 구현 공수가 늘어나며, 1인 개발자의 빠른 출시 목표에 역행함.

### 쟁점 5. MVP 5개 기능 적정성
[결론] **적정하나 '위젯'과 '알림'의 디자인 품질에 몰입할 것.**
[근거] 문장 앱의 핵심 가치는 수집보다 '노출'에 있음. TTS 기능의 우선순위를 낮추고, iOS 잠금화면 위젯의 심미성과 알림의 UX(Rich Notification) 완성도를 높이는 것이 초기 리텐션 확보에 유리함.

### 쟁점 6. 1인 개발자 4~6주 일정 현실성
[결론] **도전적이지만 가능, 단 '위젯 브릿지'가 복병.**
[근거] 단순 로직은 2주 내 완료 가능하나, RN에서 SwiftUI WidgetKit을 Config Plugin으로 연결하고 데이터 공유(App Group)를 설정하는 삽질 구간에서 1~2주가 소요될 수 있음을 대비해야 함.

시니어 크로스플랫폼 개발자로서 4인의 의견을 종합 및 비판하며, 실무 관점의 Round 2 반론을 제기합니다.

### 1. 다른 의견과 강하게 다른 점: "Expo workflow에 대한 환상과 위젯의 본질"
대부분 RN+Expo를 권장하나, **'위젯'이 킬러 피처라면 Expo Go 수준의 개발은 불가능함**을 직시해야 합니다. 
- **Opus**가 언급한 Bare Workflow(현 Prebuild)는 선택이 아닌 필수입니다. 
- 특히 iOS 위젯은 `WidgetKit`(Swift) 기반이므로, RN 레이어에서의 브릿지 구현보다 **전용 네이티브 타겟 관리 비용**을 계획에 넣어야 합니다. 단순히 '위젯이 복병'이라는 추상적 경고를 넘어, `Expo Config Plugins`를 직접 작성하거나 네이티브 코드를 만져야 하는 상황을 1인 개발자가 감당할 수 있는지 재검토가 필요합니다.

### 2. 보강해야 할 점: "ROI 중심의 피처 다이어트"
- **Codex**와 **Sonnet**이 제안한 'TTS 후순위' 및 '수동 보정 UI 강화'가 정답입니다. 
- OCR의 핵심은 인식률보다 **'오인식된 데이터를 얼마나 편하게 수정하느냐'**에 있습니다. GPT-4o 폴백은 과잉 스펙이며, 그 비용과 시간을 **검색 및 태그 시스템(Opus 제안)**에 투자하는 것이 '수집 앱' 본연의 가치에 부합합니다.

### 3. 모두가 놓친 치명적 리스크
- **위젯 데이터 동기화(App Group):** 앱 본체와 위젯 프로세스는 별개입니다. `Shared Storage`나 `App Group` 설정을 통한 실시간 데이터 동기화 이슈는 RN 초보자가 가장 많이 좌절하는 지점입니다.
- **한글 폰트 렌더링 및 레이아웃:** 명언 앱의 생명은 미학입니다. 위젯 내 한글 가독성, 장평 조절, 커스텀 폰트 적용 시 메모리 제한으로 인한 위젯 크래시 리스크가 간과되었습니다.
- **데이터 유실과 백업:** 수집 앱에서 가장 무서운 건 데이터 유실입니다. iCloud/Google Drive 연동 혹은 최소한의 JSON 내보내기 기능이 MVP에 포함되지 않으면 사용자 이탈을 막을 수 없습니다.

### 4. 합의 가능한 최소 공통분모 (Consensus)
- **기술 스택:** React Native + Expo (EAS Build 환경 필수).
- **OCR:** 플랫폼 기본 API(iOS: Vision, Android: ML Kit) 우선 활용.
- **범위:** 1차 MVP에서 TTS/GPT-4o 제외, [수집-OCR-수정UI-위젯-알림] 4개 피처 집중.
- **일정:** iOS 우선 출시 기준 6~8주, 안드로이드는 빌드 환경 안정화 후 확장.

**결론:** 기술적 타협보다 '사용자 경험의 완결성'이 중요합니다. 위젯의 네이티브 난이도를 인정하고, 부가 기능(TTS)을 쳐내어 확보한 시간으로 **폰트 미학**과 **데이터 안정성**을 확보하십시오.
